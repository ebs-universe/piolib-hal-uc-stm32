30/*
31 * This implementation is designed for multi-channel ADC sampling with
32 * relatively uniform conversion parameters. Specifically, all enabled
33 * channels are added into a single conversion group per ADC. Conversion
34 * can be triggered by some as yet unspecified sources, including by
35 * software. Each trigger might launch a single scan or a continuous
36 * scan based on the configured adc mode. Single channel conversion is
37 * not presently supported.
38 *
39 * Each conversion result is stored temporarily in the adc peripheral's
40 * state variable, and a handler function too be provided by the
41 * application is called with the ADC interface number, channel number,
42 * and a pointer to the stored result. The application must provide this
43 * function to correctly dispatch the value to the actual consumer.
44 *
45 * This interface does not support at all the following features:
46 * - Analog Watchdog
47 * - Delayed Conversion Mode
48 * - Discontinuous Conversion Mode
49 * - Reduced Resolution
50 * - DMA integration
51 * - ADF integration
52 * - Bulb, Trigger, and Plus sampling modes
53 * - Offsets
54 *
55 * Some early steps are taken to implement the following features, but
56 * applications should treat them as unsupported:
57 * - Injected Groups (some hooks included)
58 * - Oversampling
59 *
60 * Most other options are static and are hard-coded or otherwise specified
61 * at compile time only. Enhancement of the interface to allow these
62 * missing features, variable sampling rate and sampling time, simultaneous
63 * sampling, DMA, etc. may be considered if serious use cases present
64 * themselves. For the moment, applications are expected to implement
65 * their own ADC interfaces as needed for complex use cases.
66 *
67 * This implementation also does not support arbitrary channel ordering.
68 * Specifically, we restrict the channel configuration to each channel
69 * appearing at most once in the the regular sequence register, and the
70 * conversion sequence to always be a monotonic increasing channel
71 * number.
72 *
73 * Implementation Notes
74 * --------------------
75 * Most ADC config changes can only be done when the ADC is disabled.
76 * This implementation is not currently designed to handle on-the-fly
77 * changes, and the init functions are generally expected to be called
78 * exactly once during system power up.
79 *
80 * ADC Modes
81 * ---------
82 *
83 * ADC_MODE_DISABLED:
84 * ADC remains off and unconfigured. If the mode is set to disabled,
85 * this interface will be unavailable to the application, even at
86 * runtime. However, the application can use a fully independent
87 * interface, provided by iteself, though which it which can configure
88 * and use the ADC with full flexibility.
89 *
90 * ADC_MODE_SINGLE:
91 * Not presently supported by this implementation. It might later be
92 * added by using instead a dynamic conversion group with a single
93 * channel. We'll probably need to couple it with some kind of mutex
94 * or so to make sure individual calls do not clobber each other.
95 *
96 * ADC_MODE_SCAN:
97 * Scan though all enabled channels once, stop when done.
98 *
99 * ADC_MODE_CONTINUOUS:
100 * Scan though all enabled channels once, repeat when done.
101 *
102 * This chip also supports dual ADC modes, which can be used for
103 * simultaneous sampling. While we do want to support this, we're
104 * deferring it for the moment.
105 */
